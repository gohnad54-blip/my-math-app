# Технічне завдання
## Сайт-генератор математичних завдань "Задачі з параметрами"

---

## 1. Огляд продукту

**Мета продукту:** веб-застосунок, що генерує унікальні математичні завдання з розділу "Задачі з параметрами" (лінійні, квадратні, ірраціональні, тригонометричні рівняння/нерівності з параметром, графічний метод) за допомогою Gemini API, перевіряє відповіді користувача семантично та надає повний покроковий розв'язок.

**Аудиторія:** учні старших класів, абітурієнти, що готуються до ЗНО/НМТ, студенти математичних/педагогічних спеціальностей, учасники олімпіад.

**Ціннісна пропозиція:** необмежена кількість унікальних завдань (на відміну від статичних збірників), миттєва перевірка з поясненням, що еквівалентна письмовій відповіді, а не строковому збігу, три рівні складності, можливість експорту в PDF для офлайн-практики.

**Доступ:** закритий, по одному статичному паролю (приватний навчальний інструмент, не публічний продукт).

## 2. Технологічний стек з обґрунтуванням

| Шар | Технологія | Обґрунтування |
|---|---|---|
| Фреймворк | **Next.js 14 (App Router, TypeScript)** | Потрібен повноцінний бекенд + роутинг сторінок + SSR. Next.js дає API Routes "з коробки" — Gemini-ключ ніколи не потрапляє у фронтенд-бандл. Один репозиторій для фронту й бека спрощує деплой через Cursor. |
| Стилізація | **Tailwind CSS** | Швидка розробка, легко реалізувати кілька акцентних палітр через CSS-змінні/`data-theme`. |
| Формули | **KaTeX** (через `react-katex`) | Швидший за MathJax (синхронний рендер без перерахунку шрифтів), достатньо повний для шкільної/вступної математики (дроби, корені, системи, нерівності). MathJax обрано б лише за потреби рідкісних символів — тут не потрібно. |
| Бекенд-логіка | **Next.js API Routes** (`/app/api/*`) | Не потрібен окремий сервер — serverless-функції достатні для двох ендпоінтів (генерація, перевірка) + сесійного пароля. |
| Сесії/прогрес | **iron-session** (signed, httpOnly cookie, без `Max-Age`) + **sessionStorage на клієнті для лічильників** | Не потрібна повноцінна БД: дані живуть лише в межах сесії одного користувача без персистентності. iron-session шифрує і підписує cookie без зовнішнього сховища — це і є "сервер-сесія" для пароля. Лічильники прогресу (кількість розв'язаних/правильних) зберігаються в `sessionStorage` на клієнті, оскільки це чисто UI-стан, який не впливає на безпеку і не вимагає сервера. |
| Rate-limiting | **`@upstash/ratelimit` + Upstash Redis (free tier)** | Легке KV-сховище для лічильника запитів на IP/сесію, безкоштовний тір достатній для навчального проєкту. Альтернатива — in-memory `Map` у serverless-функції, але вона не переживає холодний старт/масштабування, тому обираємо Upstash. |
| Gemini API | `@google/generative-ai` SDK, модель **`gemini-2.5-flash`** | Достатньо потужна для математичних розв'язків із покроковим поясненням, підтримує `responseMimeType: "application/json"` + `responseSchema` (строгий structured output), значно дешевша й швидша за Pro-варіант, що критично при retry-механізмі (до 3 викликів на одну дію користувача). Якщо в процесі розробки якість розв'язків виявиться недостатньою для олімпіадного рівня — передбачити легке перемикання на `gemini-2.5-pro` через змінну середовища `GEMINI_MODEL`. |
| Хостинг | **Vercel (free/hobby tier)** | Нативна підтримка Next.js, нульова конфігурація деплою, інтеграція з Cursor через Git push, безкоштовний тір покриває навчальний трафік. |
| Експорт PDF | **`@react-pdf/renderer`** на сервері (API route `/api/export`) | Не вимагає headless-браузера (на відміну від Puppeteer, який важко й дорого запускати в serverless-середовищі Vercel). LaTeX-формули попередньо рендеряться в SVG через `katex.renderToString` → конвертуються в `<Image>`-елементи `@react-pdf/renderer`. |

**Чому НЕ повноцінна БД:** дані не повинні переживати закриття вкладки, користувачів не потрібно автентифікувати окрім єдиного пароля, обсяг даних на сесію — кілька чисел і короткий список хешів завдань. Будь-яка БД (Postgres, Mongo) додає інфраструктурну складність без функціональної вигоди. Якщо в майбутньому з'явиться вимога зберігати історію між сесіями — тоді доцільно додати Vercel KV/Postgres.

**Обґрунтування вибору фреймворку:** один статичний `index.html` неможливий, оскільки потрібен бекенд для безпечного зберігання Gemini-ключа, серверна перевірка пароля та генерація PDF на сервері. Next.js App Router обрано (не Pages Router), бо він дає Server Components для статичних частин UI та чіткий поділ `app/api` для бекенд-логіки в межах одного проєкту — це мінімізує конфігурацію, яку Cursor має згенерувати.

**Стейт-менеджмент:** без Redux/Zustand — досить React `useState`/`useContext` для теми оформлення (`ThemeContext`) та React Server Components + `fetch` для даних. Поточне завдання передається між `/generate` і `/task` через `sessionStorage` (ключ `currentTask`), оскільки це тимчасові дані одного кроку користувача, а не глобальний стан застосунку.

## 4. Механізм доступу (пароль)

1. На `/login` користувач вводить пароль у текстове поле (тип `password`).
2. Форма відправляє `POST /api/auth` з тілом `{ password: string }`.
3. Сервер порівнює введений пароль з `process.env.APP_PASSWORD` (env-змінна, **ніколи** не передається на клієнт).
4. При збігу — сервер створює сесію через `iron-session`: підписаний і зашифрований cookie з полем `{ authenticated: true, createdAt: number }`.
5. **Критично:** cookie встановлюється **без атрибута `Max-Age`/`Expires`** — це робить його сесійним cookie браузера, який видаляється автоматично при закритті останньої вкладки браузера (не просто вкладки сайту, а всього браузера — стандартна поведінка сесійних cookie).
6. `middleware.ts` перевіряє наявність і валідність цього cookie для всіх маршрутів `/generate`, `/task`, та всіх `/api/*` окрім `/api/auth`. При відсутності/невалідності — редірект на `/login` (для сторінок) або `401 Unauthorized` (для API).
7. Кнопка "Вийти" на головному екрані викликає `POST /api/logout`, який очищує cookie (`iron-session` `destroy()`), і клієнт додатково очищує `sessionStorage`.

**Чому не клієнтська перевірка:** клієнтський JS легко інспектувати в DevTools — пароль і будь-яка логіка порівняння на фронтенді миттєво компрометуються. Перевірка лише на сервері — обов'язкова вимога.

## 5. Категорії та логіка генерації завдань

### 5.1 Категорії (`lib/categories.ts`)
```ts
export const CATEGORIES = [
  { id: "linear", label: "Лінійні рівняння та системи з параметром" },
  { id: "quadratic", label: "Квадратні рівняння та нерівності з параметром" },
  { id: "irrational", label: "Ірраціональні рівняння та нерівності з параметром" },
  { id: "trigonometric", label: "Тригонометричні рівняння та нерівності з параметром" },
  { id: "graphical", label: "Графічний метод розв'язання рівнянь/нерівностей з параметром" },
] as const;
```

### 5.2 Рівні складності

| Рівень | Кроків розв'язку | Числові коефіцієнти | Ймовірність комбінування тем |
|---|---|---|---|
| **Базовий** | 2–4 | "охайні" (цілі, прості дроби типу 1/2, 1/3) | 90% одна тема / 10% дві теми |
| **Середній** | 4–7 | змішані (можуть бути неохайні дробові/ірраціональні числа в умові, але "чисті" в DOMAIN-аналізі) | 55% одна тема / 35% дві теми / 10% три+ теми |
| **Складний (олімпіадний)** | 7+, потребує повного дослідження усіх випадків параметра | довільні, можуть включати буквені вирази, нестандартні комбінації | 25% одна тема / 45% дві теми / 30% три+ теми |

### 5.3 Логіка випадкового комбінування тем

Користувач **не вибирає** комбінацію — лише: (а) конкретну категорію або "Будь-яка" (тоді базова тема також обирається випадково), (б) рівень складності.

Алгоритм (`lib/categories.ts`, функція `pickTopics(baseCategory, difficulty)`):

1. Якщо `baseCategory !== "random"` — вона завжди входить до набору тем як перша (гарантована).
2. Якщо `baseCategory === "random"` — випадково обирається одна з 5 категорій як базова.
3. Кидається "зважена кістка" відповідно до таблиці вище (`Math.random()` проти кумулятивних порогів), щоб визначити, скільки **додаткових** тем долучити (0, 1, або 2+).
4. Додаткові теми обираються випадково з решти 4 категорій без повторів.
5. Підсумковий список тем (1–5 елементів) передається в промпт Gemini як список, який модель має органічно поєднати в одному завданні (наприклад: квадратне рівняння, де параметр додатково обмежений тригонометричною умовою).

### 5.4 UI вибору параметрів генерації (`/generate`)

- **Селектор категорії** (`CategorySelector.tsx`): радіо-кнопки/dropdown з 6 опціями — 5 категорій + "Випадкова".
- **Селектор складності** (`DifficultySelector.tsx`): 3 кнопки-перемикачі — Базовий / Середній / Складний.
- Кнопка **"Згенерувати завдання"** — disabled під час запиту, показує `LoadingSpinner` з текстом "Генеруємо завдання…".
- При успіху — редірект на `/task`, об'єкт завдання записаний у `sessionStorage.currentTask`.

## 6. Формат відповіді та структура даних від Gemini

### 6.1 Системний промпт для генерації (`lib/prompts.ts`, функція `buildGenerationPrompt`)

Промпт обов'язково містить:

1. **Роль:** "Ти — досвідчений укладач олімпіадних задач з математики для українських учнів."
2. **Список тем** (1–5 категорій з п. 5.3) з коротким описом очікуваного типу задачі для кожної (взято зі стилю методичних матеріалів: дослідження параметра по випадках, теорема Вієта, ОДЗ, графічний метод тощо — **не копіювати конкретні приклади**, лише стиль і рівень).
3. **Рівень складності** з явним описом (кількість кроків, "охайність" чисел — див. таблицю 5.2).
4. **Вимога унікальності** (див. розділ "Інструкція щодо унікальності" нижче — пункт 6.4).
5. **Вимога LaTeX-нотації** в умові (одинарні `$...$` для inline, `$$...$$` для блокових формул).
6. **Вимога явно вказати формат відповіді** користувачу — наприклад: "Введіть відповідь у вигляді a ∈ (...)", "Введіть ціле число", "Введіть значення параметра через кому, якщо їх декілька".
7. **Сувора вимога JSON-виводу** за схемою нижче, без жодного тексту поза JSON.
8. Українська мова для всього текстового вмісту.

### 6.2 JSON-схема відповіді (Gemini `responseSchema`, дублюється в `lib/taskSchema.ts` через Zod)

```json
{
  "task_statement": "string — умова завдання з LaTeX",
  "topics": ["string"],
  "difficulty": "basic | medium | hard",
  "answer_format_hint": "string — пояснення формату відповіді українською",
  "correct_answer": "string — канонічна правильна відповідь (LaTeX або текст)",
  "solution_steps": [
    { "step_number": 1, "title": "string", "content": "string — LaTeX-пояснення кроку" }
  ],
  "coefficients_signature": "string — короткий рядок ключових чисел/букв, використаних у завданні, для дедуплікації"
}
```

Виклик Gemini виконується з параметром:
```ts
generationConfig: {
  responseMimeType: "application/json",
  responseSchema: TASK_SCHEMA // об'єкт-схема, відповідний JSON-схемі вище
}
```

### 6.3 Валідація

Після отримання відповіді JSON парситься і валідується через Zod-схему `lib/taskSchema.ts`. Якщо парсинг/валідація провалюються — спрацьовує retry-механізм (розділ 8).

### 6.4 Інструкція щодо унікальності завдань (вбудовується в кожен промпт генерації)

У системний промпт **завжди** додається такий блок (англомовний технічний текст у самому промпті допустимий, якщо це підвищує точність виконання моделлю, але user-facing контент залишається українським):

```
ВИМОГИ ДО УНІКАЛЬНОСТІ:
1. Використовуй нетипові, рандомізовані числові коефіцієнти. Уникай "шаблонних" 
   чисел (1, 2, 3, -1, 0, 5) без математичної необхідності — за можливості 
   використовуй коефіцієнти на основі поточного timestamp/seed: {randomSeed}.
2. Не повторюй словесні шаблони умови. Чергуй формулювання, наприклад:
   "Знайдіть усі значення параметра a, для яких...",
   "При яких значеннях параметра b рівняння...",
   "Дослідіть, для яких параметрів c...",
   "Визначте множину значень параметра m, при яких...",
   "Залежно від значення параметра k розв'яжіть...".
   Обери формулювання, ВІДМІННЕ від наведеного нижче списку вже використаних 
   за цю сесію: {usedPhrasingPatterns}.
3. НЕ генеруй завдання, схоже за підходом/коефіцієнтами на наведені нижче 
   сигнатури вже згенерованих у цій сесії завдань: {usedSignatures}.
   Якщо тема/рівень збігаються з уже використаними — обери інший математичний 
   прийом дослідження (інший метод розв'язання) для розрізнення.
```

Де `{randomSeed}`, `{usedPhrasingPatterns}`, `{usedSignatures}` підставляються динамічно з серверної логіки (розділ 9.3).

## 7. Логіка перевірки відповіді

### 7.1 Введення відповіді

Текстове поле (`<input type="text">`) у `TaskCard.tsx`. Вільний формат, без масок — користувач орієнтується на `answer_format_hint`. Кнопка "Перевірити" активна тільки коли поле не порожнє.

### 7.2 Запит перевірки

`POST /api/check-answer` з тілом:
```json
{
  "task_statement": "...",
  "correct_answer": "...",
  "user_answer": "...",
  "answer_format_hint": "..."
}
```

### 7.3 Промпт перевірки (`buildCheckPrompt`)

```
Ти — суворий, але справедливий перевіряючий математичних відповідей.
Умова завдання: {task_statement}
Очікуваний формат відповіді: {answer_format_hint}
Правильна відповідь: {correct_answer}
Відповідь користувача: {user_answer}

Визнач, чи є відповідь користувача математично ЕКВІВАЛЕНТНОЮ правильній

{

}
```

### 7.4 Логіка показу розв'язку

- **Якщо `is_correct === true`:** `ResultPanel` показує зелене повідомлення "Правильно! 🎉" і `reasoning`. Розв'язок (`SolutionSteps`) **не рендериться** автоматично, але присутня кнопка "Показати розв'язок", яка при натисканні монтує компонент.
- **Якщо `is_correct === false`:** `ResultPanel` показує червоне повідомлення "Неправильно" + `reasoning` + `correct_answer`. `SolutionSteps` рендериться **автоматично** одразу під результатом, але має кнопку-перемикач "Згорнути розв'язок" / "Розгорнути розв'язок" (`useState<boolean>` для toggle, як акордеон).

## 8. Retry/fallback механізм

Реалізація в `lib/gemini.ts`, функція `callGeminiWithRetry<T>`:

```ts
async function callGeminiWithRetry<T>(
  promptFn: () => Promise<string>,   // викликає Gemini SDK, повертає raw text
  schema: ZodSchema<T>,
  maxRetries = 2
): Promise<{ success: true; data: T } | { success: false; error: string }> {
  let lastError = "";
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const raw = await Promise.race([
        promptFn(),
        timeout(15000) // 15с таймаут на спробу
      ]);
      const parsed = JSON.parse(raw);
      const validated = schema.parse(parsed); // кидає ZodError при невідповідності схемі
      return { success: true, data: validated };
    } catch (err) {
      lastError = err instanceof Error ? err.message : "Unknown error";
      // короткий backoff перед наступною спробою
      if (attempt < maxRetries) await sleep(500 * (attempt + 1));
    }
  }
  return { success: false, error: lastError };
}
```

- **Усього до 3 спроб** (1 початкова + 2 retry) без участі користувача.
- Якщо всі спроби невдалі — API route повертає `{ error: "Не вдалося згенерувати завдання. Спробуйте ще раз.", code: "GENERATION_FAILED" }` зі статусом `503`.
- Фронтенд (`TaskCard.tsx`/`CategorySelector.tsx`) показує `ErrorBanner` з текстом і кнопкою **"Повторити спробу"**, яка повторно викликає той самий запит (новий повний цикл із 3 спроб).
- Таймаут на одну спробу — 15 секунд (`Promise.race` із власною `timeout`-функцією), щоб уникнути зависання UI при мережевих проблемах.

## 9. Прогрес у межах сесії

### 9.1 Що зберігається

У `sessionStorage` (ключ `progress`):
```json
{
  "solved": 0,
  "correct": 0,
  "incorrect": 0,
  "categoriesEncountered": ["linear", "quadratic"]
}
```

### 9.2 Де і чому

Виключно клієнтський `sessionStorage` — без сервера. Обґрунтування простотою: ці дані не впливають на безпеку, не потребують захисту від маніпуляції (це лише особистий лічильник користувача для самоконтролю), а `sessionStorage` природньо зникає при закритті вкладки — точно відповідає вимозі "зникає при закритті вкладки/виході" без жодної серверної логіки.

`ProgressBar.tsx` читає/оновлює цей об'єкт після кожної перевірки відповіді (`useEffect` підписаний на результат перевірки).

### 9.3 Історія завдань для дедуплікації (відмінне від прогресу!)

У `sessionStorage` (ключ `taskHistory`) зберігається масив об'єктів:
```json
[
  { "signature": "linear|basic|a=7.3,b=-2.1", "phrasingPattern": "Знайдіть усі значення параметра a, для яких" }
]
```

- Перед кожним запитом `POST /api/generate-task` клієнт надсилає поточний `taskHistory` у тілі запиту (обмежений до останніх **20 записів**, щоб не перевантажувати промпт).
- Сервер (`lib/taskHistory.ts`) форматує цей список у `{usedSignatures}` та `{usedPhrasingPatterns}` для підстановки в промпт (розділ 6.4).
- Після успішної генерації нового завдання клієнт додає `coefficients_signature` і визначене (регексом за першими 5 словами `task_statement`) `phrasingPattern` до `taskHistory` в `sessionStorage`.
- Це чисто допоміжний механізм підвищення різноманітності, **не гарантія**: фінальну унікальність забезпечує сама модель за інструкцією з розділу 6.4.

## 11. Експорт

### 11.1 Бібліотека: `@react-pdf/renderer`

Обґрунтування: працює повністю на сервері (Node.js), не вимагає headless Chrome (Puppeteer не вписується в обмеження часу виконання/розміру serverless-функцій Vercel free tier), має декларативний React-подібний API (`<Document>`, `<Page>`, `<Text>`, `<Image>`), що добре поєднується з існуючим стеком.

### 11.2 Обробка LaTeX-формул

1. На сервері (`/api/export-pdf`) для кожного LaTeX-фрагмента (з `task_statement`, `correct_answer`, кожного `solution_steps[i].content`) викликається `katex.renderToString(latex, { output: "mathml" })` → конвертується в SVG через `mathjax-node`-подібний рендерер **або** простіше: використовується пакет `katex` з `output: "html"`, який потім рендериться у зображення через `satori` + `resvg` (легкий SVG→PNG конвертер без браузера) — обрано саме цей шлях, оскільки він не потребує headless-браузера на відміну від `puppeteer`/`html2canvas`.
2. Отримані PNG/SVG зображення формул вставляються в PDF як `<Image src={base64png} />`.

### 11.3 Структура PDF

- **Сторінка 1:** заголовок "Задача з параметрами", категорія, рівень складності, умова завдання (з відрендереними формулами), `answer_format_hint`.
- **Сторінка 2+:** "Правильна відповідь:" + `correct_answer`, далі "Розв'язання:" з усіма кроками `solution_steps` послідовно (кожен крок — заголовок + текст/формули).

### 11.4 Word-експорт (опційно)

Якщо реалізація вкладається в час розробки — використати `docx` npm-пакет (бібліотека генерації `.docx` без залежності від Office). Формули вставляються як зображення (ті самі PNG, що для PDF) через `ImageRun`. Реалізується як той самий API route з параметром `?format=docx`. **Якщо ресурс часу обмежений — цей пункт відкладається на останній милстон і може бути пропущений без втрати основної функціональності.**

## 13. Безпека

1. `GEMINI_API_KEY` — лише в `.env.local`/змінних середовища Vercel, доступний тільки в API Routes (Node.js runtime), ніколи не передається у клієнтський JS-бандл (Next.js за замовчуванням не експонує неприфіксовані `*_KEY` змінні на клієнт — переконатися, що жодна змінна не названа з префіксом `NEXT_PUBLIC_`).
2. Усі звернення до Gemini відбуваються **виключно** з серверних API Routes; фронтенд звертається тільки до власних `/api/*` маршрутів.
3. `middleware.ts` перевіряє валідність сесійного cookie перед обробкою будь-якого `/api/generate-task`, `/api/check-answer`, `/api/export-pdf` запиту — повертає `401`, якщо сесія відсутня/невалідна.
4. **Rate-limiting:** через Upstash — ліміт **20 запитів на хвилину на сесійний токен** для `/api/generate-task` і `/api/check-answer`, **5 запитів на хвилину** для `/api/export-pdf`. При перевищенні — `429 Too Many Requests` із повідомленням "Занадто багато запитів, спробуйте за хвилину".
5. **Санітизація вхідних даних:** `user_answer` перед відправкою в Gemini обмежується довжиною (макс. 500 символів) і очищується від керуючих символів через `lib/sanitize.ts` (`input.replace(/[\u0000-\u001F\u007F]/g, "").slice(0, 500)`), щоб запобігти prompt injection через надмірно довгий чи спеціально сформований вхід.
6. Усі API Routes валідують вхідне тіло запиту через Zod-схеми перед використанням (запобігання некоректним/зловмисним payload).

## 14. Нефункціональні вимоги

- **Швидкість відгуку:** генерація завдання — очікувано 3–8 секунд (зважаючи на `gemini-2.5-flash` + можливі retry). Під час очікування — `LoadingSpinner` з текстом і прогрес-індикатором (анімований, без точного відсотка, оскільки час непередбачуваний).
- **Адаптивність:** повна підтримка mobile (320px+), tablet, desktop через Tailwind breakpoints.
- **Обробка помилок мережі:** усі `fetch`-запити обгорнуті в `try/catch`; при втраті з'єднання — `ErrorBanner` "Немає з'єднання з сервером. Перевірте інтернет і спробуйте ще раз" з кнопкою повтору.
- **Доступність (a11y):** усі інтерактивні елементи мають `aria-label`, поле відповіді — `aria-describedby` на `answer_format_hint`, фокус-кільця видимі (`focus-visible:ring`).
